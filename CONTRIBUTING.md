# Contributing

Contributions are welcome when they make the skill more focused, practical, and reliable for modular architecture implementation.

## Contribution principles

A good contribution should:

- improve the agent's ability to ask useful questions before work begins
- improve targeted project scanning rather than broad project-wide behavior
- improve modular architecture guidance around boundaries, contracts, dependency direction, and source of truth
- improve robustness guidance around validation, errors, idempotency, migrations, observability, and tests
- improve implementation safety through smaller phases, verification, and stop conditions
- keep examples generic and reusable

Avoid contributions that:

- turn the skill into a global style guide
- add long generic checklists that apply to every task
- encourage broad rewrites before evidence is inspected
- prefer fashionable patterns over project evidence
- introduce domain-specific assumptions
- remove the requirement to ask targeted questions before work begins

## Pull request checklist

Before opening a pull request:

- Read `SKILL.md` end to end.
- Confirm the activation rules still prevent global application.
- Confirm the workflow still asks questions before scanning or editing.
- Confirm examples remain generic and implementation-focused.
- Confirm new guidance is actionable and verifiable.
- Update README documentation when behavior changes.

## Documentation style

Use concise, direct language. Prefer concrete instructions over abstract theory. Every rule should help an agent produce better code or better implementation plans.
